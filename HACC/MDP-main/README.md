# MDP

This repository contains three versions:

1. `m_axi_gmem` linked to `HBM[0]`
2. `m_axi_gmem0`, `m_axi_gmem1`, `m_axi_gmem2`, `m_axi_gmem3`, `m_axi_gmemQ` linked to `HBM[0]`
3. `m_axi_0` and `m_axi_1` linked to `HBM[0]` and `HBM[1]`, respectively

## Host application 
The host application can be compiled from the repository root without moving into the project folders.

Before building, make sure the Vitis and XRT environment variables are available. Here are some possible solutions

```bash
export XILINX_VITIS=/home/DIRECTORY/2025.1/Vitis
export XILINX_XRT=/opt/xilinx/xrt
```

For `host_application_v1_v2`:

```bash
cmake -S host_application_v1_v2 -B build/host_application_v1_v2
cmake --build build/host_application_v1_v2 -j
```

For `host_application_v3`:

```bash
cmake -S host_application_v3 -B build/host_application_v3
cmake --build build/host_application_v3 -j
```

The generated executable is named `bambu_application` and will be created under the selected build directory.

The host application can be run by:

 ```bash
export XCL_EMULATION_MODE=hw_emu
export XRT_VERBOSE=3
export LD_LIBRARY_PATH=/opt/xilinx/xrt/lib:/usr/lib/x86_64-linux-gnu:$LD_LIBRARY_PATH
```

For `host_application_v1_v2`:

 ```bash

/home/DIRECTORY/host_application_v1_v2/build/bambu_application \
/home/DIRECTORY/xo/panda.xclbin \
/home/DIRECTORY/data/input.data \
/home/DIRECTORY/data/check.data
```
For `host_application_v3`:

 ```bash
/home/DIRECTORY/host_application_v3/build/bambu_application \
/home/DIRECTORY/xo/panda.xclbin \
/home/DIRECTORY/data/input.data \
/home/DIRECTORY/data/check.data
```

If you need hw_emu you should export it, otherwise simply do not use this line.

## Create the `.xo`

To create the `.xo`, put `bfs_translator.v`, `bfs.v`, and `panda_libtech.v` under a `src` folder.

Expected structure:

```text
version_3/
├── src/
│   ├── bfs.v
│   ├── bfs_translator.v
│   └── panda_libtech.v
└── script_to_xo.tcl
```

Run:

```bash
vivado -mode batch -source version_X/script_to_xo.tcl
```

After this, run the `xo_to_xclbin` script in the terminal to create the `.xclbin` (binary executable).

## Important flags

- `-t hw_emu`: hardware emulation only
- `-t hw`: implementation flow (still generates a `.xclbin` file)

## Signals & Debug

You have two options to inspect signals:

1. Add `-g` during the link phase (`xo` to `xclbin`).
   You can inspect only `m_axi` and `s_axi` signals.
   In Vivado, navigate to:
   `pfm_top_wrapper -> pfm_top_i -> pfm_dynamic_inst -> panda_1`,
   then add signals to the wave window.

2. To inspect all signals, use option 1 and add `xrt.ini` plus a debug `.tcl` script.
   Launch host and kernel with:

   ```bash
   export XCL_EMULATION_MODE=hw_emu
   export XRT_VERBOSE=3
   export LD_LIBRARY_PATH=/opt/xilinx/xrt/lib:/usr/lib/x86_64-linux-gnu:$LD_LIBRARY_PATH
   ```
   For version 1 and 2
   
   ```bash
   ./host_application_v1_v2/build/bambu_application \
   ./host_application_v1_v2/xo/panda.xclbin \
   ./data/input.data \
   ./data/check.data
   ```

   For version 3
   
   ```bash
   ./host_application_v3/build/bambu_application \
   ./host_application_v3/xo/panda.xclbin \
   ./data/input.data \
   ./data/check.data
   ```

   Optionally set:

   ```bash
   export XRT_INI_PATH=/path/to/xrt.ini
   ```

   If `xrt.ini` is in the same launch directory, `XRT_INI_PATH` may not be required.
   Debug files are available in `debug_all/`.

To generate kernel schematics, add `-s`.
